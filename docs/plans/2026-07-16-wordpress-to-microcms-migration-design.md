# WordPress記事のmicroCMS移行設計

## 目的

旧WordPressサイトで公開中の投稿30件を、本文画像を含めてmicroCMSの`news`エンドポイントへ移行する。既にmicroCMSで整形済みの重複3件は維持し、残り27件を下書きとして登録する。移行後は旧WordPressを停止しても記事と画像を表示できる状態にする。

## 対象

- 移行元: `https://jqit.co.jp/wp-json/wp/v2/posts`
- 移行先: microCMS `news`エンドポイント
- WordPress公開投稿: 30件
- 既存microCMS記事: 4件
- 重複として維持する記事:
  - 社員総会 → `soukai-202604`
  - 社員の交流 → `kouryu-202604`
  - ISO27001（ISMS）認証取得のお知らせ → `iso27001-isms-20260306`
- 新規下書き登録: 27件

`site-renewal-2026`はWordPress移行対象とはせず、microCMS既存記事として維持する。

## 採用方式

再実行可能なNode.js移行スクリプトを実装する。WordPress REST APIから投稿を取得し、画像をmicroCMSメディアへアップロードして本文URLを書き換え、microCMSへ下書き登録する。

手動登録は作業漏れと再現性の問題があるため採用しない。旧WordPress画像URLを本文に残す方式は旧サイト停止後に画像切れするため採用しない。

## データマッピング

| WordPress | microCMS | 変換 |
|---|---|---|
| 投稿ID | コンテンツID | `wp-{投稿ID}`。重複3件のみ既存IDへ対応付ける |
| `title.rendered` | `title` | HTMLエンティティをデコードし、不要なタグを除去 |
| `date` | 公開日 | WordPressの日時を維持できるフィールド／API方式を事前検証する |
| カテゴリ | `category` | 内容に応じて「お知らせ」「イベント」を割り当てる。判定不能時は「お知らせ」 |
| `content.rendered` | `content` | WordPress固有マークアップを整理し、安全なHTMLへ変換 |
| アイキャッチ | 本文先頭画像 | microCMSメディアへアップロードして本文先頭へ配置 |
| 本文画像 | `content`内画像 | microCMSメディアへアップロードし、URLを書き換える |

現行microCMSスキーマでは画像専用フィールドが定義されていない場合があるため、サイト実装が対応済みの`content`内画像を標準とする。

## 処理フロー

1. WordPress REST APIから公開投稿30件を取得する。
2. microCMSの既存コンテンツを取得する。
3. 重複3件をスキップし、旧URLと既存IDの対応を記録する。
4. 各投稿のアイキャッチと本文画像を収集する。
5. 画像をダウンロードし、形式・サイズ・HTTPステータスを検証する。
6. microCMS Management APIへ画像をアップロードする。
7. WordPress固有HTMLを整理し、画像URLをmicroCMS URLへ置き換える。
8. 27件を下書きとしてmicroCMSへ登録する。
9. 旧URL、新コンテンツID、処理状態を移行マニフェストへ出力する。
10. 全件を検証し、確認後に別操作で公開する。

## 再実行性と安全性

- 既存コンテンツIDがある場合はデフォルトで上書きしない。
- 画像の移行結果をマニフェストへ保存し、同一画像の再アップロードを避ける。
- `--dry-run`ではWordPress取得、変換、件数、画像一覧だけを確認し、microCMSを変更しない。
- `--apply`を指定した場合のみ外部データを変更する。
- エラー発生時は記事単位で失敗理由を記録し、途中から再実行できるようにする。
- APIキーや認証情報はログ・マニフェストへ書き出さない。

## エラー処理

- WordPress投稿取得失敗: 移行開始前に終了する。
- 画像取得失敗: 該当記事を失敗扱いにし、画像切れのまま登録しない。
- 5MB超過画像: 圧縮対象として報告し、自動登録を止める。
- microCMSアップロード失敗: 記事を登録せず、再実行可能な状態を維持する。
- microCMS記事登録失敗: HTTPステータスと記事IDを記録するが、秘密値や本文全体は出力しない。

## 検証

- dry-run結果が「30件取得、3件スキップ、27件作成予定」になること。
- 27件すべてのタイトル、日付、本文が空でないこと。
- 本文内に`jqit.co.jp/hp/wp-content/uploads`が残らないこと。
- 全画像URLがmicroCMS AssetsでHTTP 200を返すこと。
- microCMS下書き一覧で27件を確認できること。
- プレビューまたはローカル取得で本文が安全に表示されること。
- 既存4件の内容が変更されていないこと。

## 公開とURL移行

内容確認後に27件を公開する。旧WordPress URLから新しい`/news/{contentId}`への301リダイレクトは移行マニフェストを元に別途実装する。重複3件は既存の整形済み記事へリダイレクトする。

## 対象外

- WordPress固定ページと採用ページの移行
- 問い合わせフォームのResend設定
- DNS切替
- microCMS記事の一括公開操作（下書き確認後に実施）
